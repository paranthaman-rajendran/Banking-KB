# ISO 20022

The dominant payment messaging standard. XML-based (with JSON renderings emerging), structured around business components (parties, accounts, amounts, instruction identifiers, remittance information).

> Sub-domain: `Payments / Standards`
> Reference: ISO 20022 Repository (iso20022.org) + ISO 20022 Registration Authority (SWIFT)

---

## Message Family Naming

`<business_area>.<message_id>.<variant>.<version>`

Example: `pacs.008.001.10` — Payments Clearing & Settlement, FI-to-FI customer credit transfer, variant 001, version 10.

| Business Area | Code | Used For |
|---|---|---|
| Payments — Clearing & Settlement | `pacs` | Inter-bank credit transfer, return, recall |
| Payments — Initiation | `pain` | Customer-to-bank initiation (e.g., corporate payroll) |
| Cash Management | `camt` | Statements, notifications, reports, intra-day |
| Securities Settlement | `sese` | Securities settlement instructions (out of Payments scope) |
| Cards | `caaa`, `catp`, etc. | Card acceptance |
| Trade Services | `tsmt` | Trade finance |

For Payments KB, the families that matter are `pacs`, `pain`, and `camt`.

## Structured Data Wins

ISO 20022 brings structured fields where MT had free text:

| Field | MT | MX (ISO 20022) |
|---|---|---|
| Ordering Customer | 50K (1×35 free text) | Structured name, address, ID, country |
| Remittance Info | 70 (4×35 free text) | Structured remittance — invoice ref, RFE codes, line items |
| Charges | 71A/71F/71G | Structured ChargeBearer + ChargeAmount per agent |
| UETR | gpi extension | Native UETR field |
| Purpose | 26T / 70 | Structured purpose code (ISO 20022 ExternalPurpose code list) |
| Regulatory Reporting | Free text | Structured DebtRegulatoryReporting / CredRegulatoryReporting |

This structure makes the KB query model materially better — you can index against typed fields, not free text.

## CBPR+ — SWIFT's Cross-Border ISO 20022 Profile

**CBPR+** (Cross-Border Payments and Reporting Plus) is the SWIFT-managed profile for cross-border MX messages. It defines:
- Mandatory / optional fields per message
- Network validation rules
- Translation rules from legacy MT to MX (one-way: MT → MX is lossy; MT cannot fully express MX richness)

Co-existence (MT + MX accepted) ends in **November 2025** for cross-border payments. From then, MX-only.

---

## HVPS+ — High-Value Payment Systems Profile

Similar discipline for domestic high-value RTGS systems (TARGET2, FedWire, BoE CHAPS). Each operator publishes its own usage guidelines on top of HVPS+.

## Chunking Strategy (for ingestion)

ISO 20022 spec docs are large XML schemas with many constraints. Recommended chunking:
- Per **message type** as the parent unit.
- Per **field group** (e.g., `Debtor`, `Creditor`, `ChargesInformation`) as sub-chunks.
- Validation rules as separate searchable chunks linked to the field they constrain.

Avoid fixed-size chunking — XSD structure is the natural boundary.
