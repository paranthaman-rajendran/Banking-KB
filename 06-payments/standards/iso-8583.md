# ISO 8583 — Card Message Standard

The card industry's messaging standard. Used by Visa, Mastercard, RuPay, Amex, Diners, JCB, UnionPay and the ATM / POS / e-commerce flows. Older than ISO 20022 and structured very differently (bitmap-driven, binary).

> Sub-domain: `Payments / Cards`
> Reference: ISO 8583 (1987, 1993, 2003 versions; schemes use customized profiles)

---

## Message Anatomy

```
MTI (4 chars) + Primary Bitmap (64 bits) + [Secondary Bitmap (64 bits)] + Data Elements
```

| Section | Purpose |
|---|---|
| MTI | Message Type Indicator (e.g., 0100, 0200, 0210) |
| Primary Bitmap | Indicates which of fields 1–64 are present |
| Secondary Bitmap | Indicates fields 65–128 (if needed; flagged by bit 1 in primary) |
| Data Elements | Variable length, per bitmap |

The bitmap means messages are compact — only present fields are transmitted.

---

## MTI Decoding

`MTI = position1 . position2 . position3 . position4`

| Position | Meaning |
|---|---|
| 1 | ISO version (0=1987, 1=1993, 2=2003) |
| 2 | Message class (1=Authorization, 2=Financial, 4=Reversal, 5=Reconciliation, 6=Administrative, 7=Fee Collection, 8=Network Management, 9=Reserved) |
| 3 | Message function (0=Request, 1=Request response, 2=Advice, 3=Advice response, 4=Notification, etc.) |
| 4 | Message origin (0=Acquirer, 1=Acquirer repeat, 2=Issuer, 3=Issuer repeat, 4=Other, 5=Other repeat) |

### Common MTIs
- `0100` — Authorization request (e.g., online card purchase auth)
- `0110` — Authorization request response
- `0200` — Financial transaction request (e.g., ATM withdrawal)
- `0210` — Financial transaction request response
- `0400` — Reversal request
- `0410` — Reversal response
- `0800` — Network management (echo, sign-on, key change)
- `0810` — Network management response

## Scheme Profiles

Each scheme defines its own ISO 8583-derived profile:

| Scheme | Profile |
|---|---|
| Visa | VisaNet (V.I.P.) — BASE I / BASE II |
| Mastercard | Mastercard CIS / IPM (clearing) / Banknet |
| RuPay | NPCI ISO 8583 profile + RuPay extensions |
| Amex | Global Direct |
| JCB | JCB ISO 8583 |
| UnionPay | UPI (UnionPay International) |

Schemes deviate from the base standard. A card message has to be parsed against the **scheme's profile**, not generic ISO 8583.

---

## EMV Layered on Top

For chip-card / contactless / 3DS, EMV-encoded data (cryptograms, application IDs, terminal verification results) is carried inside DE 55 as TLV. This carries the heavy lifting for present-card cryptographic verification.

---

## 3-D Secure 2.x

For card-not-present (e-commerce), 3DS authenticates the cardholder before the auth message is built. 3DS uses JSON (not ISO 8583) end-to-end between MPI → DS → ACS. The result (CAVV, ECI) is then folded back into the ISO 8583 auth (`0100`) DE 48 or scheme-specific.

---

## Tokenization

PCI scope reduction via tokenization. Two flavors:
- **Network tokens** — issued by the card scheme; replace the PAN in storage and transmission.
- **Acquirer / PSP tokens** — issued by the acceptance side; valid within a merchant or PSP only.

KB content on tokens should call out which kind, since the operational and compliance footprint differs.
